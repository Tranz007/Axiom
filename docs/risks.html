Agent risk

AI agents fail differently than scripts.

They can misunderstand intent, use the wrong environment, find credentials, take irreversible actions, or claim success without proof. Axiom exists to make dangerous capabilities explicit before an agent builds.

Environment

Wrong target

An agent may touch production when the human expected a dev, staging, or local environment.

Authority

Too much access

A token or API key may allow more than the agent actually needs for the task.

Proof

False confidence

Generated code, tests, or status reports can look plausible while hiding missing checks or unsafe behavior.

Real incidents show the shape of the risk.

Public reports have described AI coding agents deleting production data, acting during code freezes, using broad credentials, and fabricating or misreporting recovery state. Axiom should not sensationalize those stories, but it should learn from the pattern.

The common pattern

  • The agent had a goal but not a durable contract.
  • Dangerous actions were possible from the available environment.
  • Approval, environment, and data boundaries were not enforced at the application contract level.
  • The human saw the damage after the action, not before.

How Axiom helps

Axiom moves risky behavior out of chat instructions and into a checked contract: what the app can touch, what actions exist, when approval is required, what brokers may execute, what is denied, and what must be audited. 

capability delete_customer_data
  policy:
    require_approval if production_target
      or irreversible_action

    deny if approval_missing
      or backup_not_verified
      or agent_requested_broad_delete

  audit:
    policy_decision
    approval_state
    never log raw_customer_record
Before code

Declare capabilities

Agents should not invent new data access or destructive actions while implementing a feature.

Before action

Require approval

Production changes, external effects, sensitive disclosures, and irreversible operations need explicit gates.

After action

Audit safely

Humans need a durable record of what happened without storing raw secrets, payment data, or private records.

What Axiom does not promise

Axiom is not a backup system, sandbox, deployment firewall, or production permission manager by itself. It is the application contract layer that helps humans and agents define risky behavior before implementation.

Still required

  • separate dev, staging, and production environments
  • least-privilege credentials
  • backups stored away from the thing being deleted
  • human review for high-risk changes
  • runtime and infrastructure controls
