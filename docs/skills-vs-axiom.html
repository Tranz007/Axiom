Agent Skills, Markdown Guides, And Axiom

Markdown can guide an agent. Axiom should constrain a system.

Agent skills, AGENTS.md files, and reusable Markdown playbooks are becoming a normal way to teach agents how to work. Axiom should respect that trend while solving a different problem: turning authority, policy, approval, disclosure, effects, and audit into executable contracts.

The trend is real

The agent ecosystem is converging on lightweight instruction packages. Claude Code skills use a SKILL.md file plus optional scripts and resources. GitHub Copilot supports agent skills that are folders of instructions, scripts, and resources. Codex reads AGENTS.md files so projects can provide persistent operating guidance.

That is a good direction. Markdown is cheap, portable, reviewable, and easy for teams to write. It is likely to become the README layer for agent behavior.

Axiom should not compete with skills as a better notebook. It should compile the parts of a system that cannot safely remain as advice.

What Markdown skills are good at

Skills and guides shine when they describe process
  • How to run a project
  • How to review code
  • How to format a document
  • How to follow team conventions
  • How to use a tool correctly
They are weaker when they must enforce authority
  • Whether an agent may touch sensitive data
  • Whether an output may leave a broker
  • Whether an approval is bound to the right destination
  • Whether a generated route violates policy
  • Whether audit logs omit raw secrets

The security problem with advice-only layers

A skill can say, never expose raw customer records. That helps. But it does not prove that an API route cannot return raw records, that a future agent did not add a broad search endpoint, or that an approval screen is binding the approved destination to the actual effect.

This is the core limitation: Markdown can express a promise, but it usually cannot enforce the promise across generated code, runtime boundaries, and deployment changes.

Axiom should be judged by whether it catches drift that a well-written skill merely warned about.

A concrete example

A skill might instruct an agent:

When building customer support tools, never expose raw identity records to an untrusted agent.
Use masked summaries unless a human explicitly approves disclosure.

That is useful guidance. The Axiom version should turn the same promise into structure:

data_class identity_record
  sensitivity high
  disclosure raw: forbidden_to untrusted_agent
  disclosure summary: allowed_if task_scoped

capability answer_support_question
  caller support_agent: untrusted
  reads identity_record through broker identity_broker
  output summary_only
  approval required_if destination_unknown

invariant no_raw_identity_to_agent
  forbid:
    route returns identity_record.raw to untrusted_agent
    audit includes identity_record.raw

The important difference is not syntax. The difference is that a compiler can generate policy tests, route guard checks, broker contracts, approval UI requirements, and audit assertions from the Axiom source.

Axiom should use skills, not dismiss them

The strongest version of Axiom would likely ship with skills. A SKILL.md can teach an agent how to author Axiom files, run the compiler, interpret policy failures, and update generated targets without bypassing the source graph.

In that relationship, skills are the agent-facing operating manual. Axiom is the system-facing contract. One helps the agent do good work. The other limits what the resulting system is allowed to do.

The claim Axiom has to earn

Axiom is more than Markdown only if it produces enforceable outputs:

  • Policy matrices that can be tested
  • Generated route guards that deny out-of-scope capabilities
  • Broker contracts that prevent raw data leakage
  • Approval flows bound to actor, destination, data class, and effect
  • Audit schemas that record decisions without leaking sensitive payloads
  • Verification reports showing generated code has not drifted from intent

If Axiom cannot generate those artifacts, developers should use Markdown skills, ordinary docs, and tests instead. The honest bar is high.
