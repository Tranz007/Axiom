Why Axiom exists

AI coding agents are good at moving fast. That is also the risk. If an app handles private data, sends messages, deletes records, calls tools, or submits forms, the agent needs clear limits before implementation starts.

Intent

Purpose is source

Product intent becomes part of the program, so generated code can be checked against what the system promised to do.

Access

No broad access by default

Agents get narrow capabilities, not open-ended permission to search records, export data, or call any tool they find.

Policy

Models do not decide access

LLMs can help build and explain, but final allow, deny, and approval decisions are deterministic software.

Human installs. Agent uses.

Axiom OS is the local, open-source-oriented path for people building with AI coding agents. The first workflow stays simple: the human adds Axiom to the project, and the agent uses it to find the contract, check policy, run simulations, and take the next implementation step.

What about agent skills?

Markdown skills, AGENTS.md files, and reusable agent guides are becoming the normal way to teach agents repeatable workflows. Axiom should complement them, not pretend they are unimportant.

Guidance is not enforcement

Skills tell an agent how to behave. Axiom turns data access, tool use, approvals, effects, and audit into compiler-checked and runtime-enforced contracts.

Why add another layer?

Because the complexity already exists in agent-facing systems. It is scattered across middleware, schema validators, UI copy, policy files, tests, audit code, threat models, and memory. Axiom earns its place only when it turns scattered safety rules into executable source.

For skeptical developers

Axiom is not for every app. It is for software where agents, sensitive data, approval, external effects, and auditability collide.
